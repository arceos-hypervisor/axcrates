# 多仓库协同开发方案

> 本文档是 `组件开发及管理规范.md` 的配套操作指南，专注于**多仓库协同的具体执行方案**。
> 
> 关于开发规范、分支规划、版本号规范等，请参考 [组件开发及管理规范.md](组件开发及管理规范.md)。

---

## 一、方案概述

### 1.1 背景与目的

ArceOS Hypervisor 采用**组件化架构设计**，组件是功能相对独立且可独立测试的实体。为了规范组件的开发、管理和发布流程，特制定本协同开发方案。

### 1.2 仓库架构

我们的组件**实际分布在各个独立的子仓库中**，每个组件都是一个独立的 Git 仓库。同时，我们提供了 **axcrates** 仓库作为统一的管理入口，通过 Git Submodule 关联所有组件。**因此，有两种开发方式可选，来处理组件的开发过程：**

| 方式 | 适用场景 | 特点 |
|------|----------|------|
| **独立子仓库开发** | 单组件修改 | 直接克隆子仓库，轻量快捷 |
| **axcrates 统一仓库开发** | 跨组件协作 | 通过主仓库统一管理所有组件 |

### 1.3 架构关系图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         组件分布与协作架构                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │

│   │ arm_vcpu     │◄───────────────────────►│  .gitmodules         │    │
│   │ arm_vgic     │◄───────────────────────►│  scripts/            │    │
│   │ axaddrspace  │◄───────────────────────►│  ├── check.sh        │    │
│   │ axdevice     │◄───────────────────────►│  ├── version.sh      │    │
│   │ axdevice_base│◄───────────────────────►│  └── publish.sh      │    │
│   │ axhvc        │◄───────────────────────►│                      │    │
│   │ axvcpu       │◄───────────────────────►│  arm_vcpu/           │    │

│   └──────────────┘                         └──────────────────────┘    │

│        │         Git Submodule 关联               │                    │
│        └──────────────────────────────────────────┘                    │
│                                                                         │
│   开发方式选择：                                                         │
│   • 方式一：直接克隆子仓库（如 arm_vcpu）独立开发                        │

│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```
- **独立子仓库**：每个组件（如 arm_vcpu、axvcpu）都有自己的独立 Git 仓库，可以独立开发、测试、发布
- **axcrates 主仓库**：通过 Git Submodule 引用所有组件，提供统一的版本管理和协同脚本


### 1.4 核心开发原则

基于 [组件开发及管理规范.md](组件开发及管理规范.md)，我们遵循以下核心原则：

| 原则 | 说明 | 实践方式 |
|------|------|----------|
| **版本统一** | 所有组件使用相同版本号，便于管理和追踪 | axcrates仓库中 `version.sh` 批量管理所有组件版本号 |
| **分支隔离** | main 分支保护（需 PR），dev 分支开放（可直接推送） | PR 流程 vs 直接推送 |
| **依赖锁定** | 主仓库记录子模块具体 commit，确保可复现构建 | `.gitmodules` + submodule |
| **独立演进** | 子仓库可独立开发测试，不受主仓库影响 | 直接克隆子仓库开发 |
| **代码质量** | 所有组件必须通过 CI 检查（fmt、clippy、doc、test） | `scripts/check.sh` 本地检查 |
| **文档完整** | 公开发布的组件必须具备完整的 rustdoc 文档 | `cargo doc` 生成文档 |

### 1.5 分支与版本对应关系

在 [组件开发及管理规范.md](组件开发及管理规范.md) 中我们规定了，组件仓库分支与版本必须强制对应，以便统一管理！

## 二、开发环境准备

### 2.1 配置 Git 环境

```bash
# 配置全局用户名和邮箱（如未配置）
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# 配置 SSH 密钥（用于访问 GitHub）
# 如已有密钥可跳过
ssh-keygen -t ed25519 -C "your.email@example.com"
cat ~/.ssh/id_ed25519.pub
# 将公钥添加到 GitHub: Settings -> SSH and GPG keys -> New SSH key
```

### 2.2 安装 Rust 工具链

```bash
# 安装 Rust（如已安装可跳过）
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env

# 安装所需目标架构
rustup target add aarch64-unknown-none-softfloat
rustup target add x86_64-unknown-none
rustup target add riscv64gc-unknown-none-elf

# 安装必要组件
rustup component add rust-src llvm-tools rustfmt clippy
```

## 三、开发工作流详解

### 3.1 工作流选择决策树

```
开始开发
    │
    ├──► 只修改单个组件？
    │       │
    │       ├──► 是 ──► 选择模式 A：独立子仓库开发
    │       │
    │       └──► 否 ──► 需要修改多个组件？
    │               │
    │               ├──► 是 ──► 选择模式 B：主仓库协同开发
    │               │
    │               └──► 否 ──► 只需要测试？
    │                       │
    │                       ├──► 是 ──► 选择模式 B：主仓库协同开发
    │                       │
    │                       └──► 否 ──► 咨询维护者
    │
    └──► 发布版本？
            │
            ├──► 预览版 ──► 使用 version.sh + commit.sh + tag.sh
            │
            └──► 稳定版 ──► 使用 version.sh + commit.sh + tag.sh + PR
```

### 3.2 独立子组件开发

对于每个独立的组件，我们已经在其对应的仓库中配备了完整的 CI/CD 以及本地测试脚本文件。当需要修改只修改单个组件，不涉及其他组件的 API 变更时，可以直接在独立子组件仓库中开发及测试。

#### 3.2.1 克隆子仓库

```bash
# 以 axvcpu 为例
git clone git@github.com:arceos-hypervisor/axvcpu.git
cd axvcpu

# 查看当前分支
git branch -a
```

#### 3.2.2 创建功能分支

所有子仓库的 `main` 或 `master` 分支受保护，**无法直接推送代码**。所有开发必须在功能分支上进行，通过 Pull Request 合并。

```bash
# 获取最新代码
git checkout dev
git pull origin dev

# 基于 dev 创建功能分支
git checkout -b feature/add-new-api

# 分支命名规范：
# - feature/xxx  - 新功能
# - fix/xxx      - Bug 修复
# - refactor/xxx - 代码重构
# - docs/xxx     - 文档更新
```

#### 3.2.3 修改代码

```bash
# 编辑代码...
# vim src/lib.rs

# 本地构建验证
cargo build
```

#### 3.2.4 代码检查

使用 `scripts/check.sh` 检查代码规范：

```bash
# 通过脚本
bash scripts/check.sh

# 或手动执行检查
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
```

**检查项目**：
1. 代码格式（`cargo fmt`）
2. 构建检查（`cargo build`）
3. Clippy 检查（`cargo clippy`）
4. 文档构建（`cargo doc`）

#### 3.2.5 本地测试

**单元测试**：
```bash
# 运行单元测试
cargo test --all-features

# 运行指定测试
cargo test test_name
```

**集成测试**：
```bash
# 使用 test.sh 执行集成测试
bash scripts/test.sh
```

#### 3.2.6 提交变更

```bash
# 查看变更
git status
git diff

# 添加变更
git add .

# 提交（使用约定式提交格式）
git commit -m "feat: add new virtualization API

This commit adds support for new vCPU features:
- Add vCPU state save/restore API
- Support for new interrupt handling

Closes #123"

# 推送到远程
git push origin feature/add-new-api
```

#### 3.2.7 创建 Pull Request

在各组件仓库页面点击 "Compare & pull request" 创建 PR：

**推送到 dev 分支（预览版迭代）**：
```
GitHub 操作：
1. 打开 https://github.com/arceos-hypervisor/axvcpu
2. 点击 "Compare & pull request"
3. base: dev ← compare: feature/add-new-api
4. 填写 PR 标题和描述
5. 请求 review（可选，dev 分支可快速合并）
6. 合并 PR
```

**推送到 main 分支（稳定版）**：
```
GitHub 操作：
1. 打开 https://github.com/arceos-hypervisor/axvcpu
2. 点击 "Compare & pull request"
3. base: main ← compare: feature/add-new-api
4. 填写详细的 PR 描述
5. 确保 CI 检查通过
6. 请求至少 1 人 review
7. 合并 PR（需 maintainer 权限）
```

#### 3.2.8 通知主仓库更新

```bash
# PR 合并后，通知主仓库维护者更新 submodule
# 提供以下信息：
# - 组件名称：axvcpu
# - 更新的分支：dev 或 main
# - 变更简述：添加了新的 vCPU API
```

### 3.3 多组件协同开发

多组件协同开发适用于需要同时修改多个组件，或者跨组件 API 变更等情况。为此，我们创建了 https://github.com/arceos-hypervisor/axcrates 统一仓库，通过 SUBMODULE 来管理所有组件。

与独立开发的最大区别，axcrates 提供了一系列**协同开发脚本**，可以**统一管理所有组件**，无需进入每个组件目录执行 git 命令。
| 脚本 | 功能 | 示例 |
|------|------|------|
| `checkout.sh` | 批量切换分支 | `bash scripts/checkout.sh all dev` |
| `check.sh` | 批量代码检查 | `bash scripts/check.sh all` |
| `commit.sh` | 批量提交组件 | `bash scripts/commit.sh axvcpu "feat: xxx"` |
| `push.sh` | 批量推送组件 | `bash scripts/push.sh all` |
| `sync.sh` | 批量同步子模块 | `bash scripts/sync.sh all dev` |
| `version.sh` | 批量更新版本 | `bash scripts/version.sh all 0.2.0` |
| `tag.sh` | 批量创建标签 | `bash scripts/tag.sh all v0.2.0` |

#### 3.3.1 获取 axcrates 仓库

```bash
# 克隆主仓库（包含所有子模块）
git clone --recurse-submodules git@github.com:arceos-hypervisor/axcrates.git
cd axcrates

# 切换到 dev 分支
git checkout dev
```
#### 3.3.2 批量切换组件分支

使用 `scripts/checkout.sh` 批量管理组件分支：

```bash
# 切换所有组件到 dev 分支
bash scripts/checkout.sh all dev

# 切换所有组件到 main 分支
bash scripts/checkout.sh all main

# 切换单个组件到 feature 分支
bash scripts/checkout.sh axvcpu feature/new-api
```

**脚本功能**：
- 自动检测当前分支，如在目标分支则跳过
- 自动从远程获取并创建跟踪分支
- 检测未提交变更并提示
- 汇总显示切换结果

#### 3.3.3 确定修改范围

```bash
# 分析依赖关系，确定需要修改的组件
# 示例：修改 axvcpu API，影响 arm_vcpu 和 x86_vcpu

# 查看组件依赖树
cd axvcpu && cargo tree
cd ../arm_vcpu && cargo tree | grep axvcpu
cd ../x86_vcpu && cargo tree | grep axvcpu
```

#### 3.3.4 修改组件代码

开发阶段进入各组件目录修改代码。此时所有组件已在 dev 分支（由 3.3.2 设置），开发流程与 3.2 独立子组件开发类似，但是可使不用在组件中直接提交代码，而是在编辑多个组件之后，统一使用 `check.sh`、`commit.sh` 批量处理多个组件。

**单组件处理**：

```bash
# 进入组件目录
cd axvcpu

# ... 编辑代码 ...
# ... 测试 ...
# ... CHECK ...

# 使用 commit.sh 提交并推送（推荐）
bash scripts/commit.sh axvcpu "refactor: update vCPU API for better abstraction"

# 或者手动提交
git add .
git commit -m "refactor: update vCPU API for better abstraction"
git push origin feature/refactor-api
```

**多组件批量处理**：

```bash
# 场景：修改了 axvcpu API，需要适配 arm_vcpu 和 x86_vcpu

# 1. 在 axvcpu 开发并提交
cd axvcpu
# ... 编辑代码 ...
# ... 测试 ...
# ... CHECK ...
# ... 提交 ...
bash scripts/commit.sh axvcpu "refactor: update vCPU API"

# 2. 在 arm_vcpu 适配并提交
cd ../arm_vcpu
# ... 编辑代码 ...
# ... 测试 ...
# ... CHECK ...
# ... 提交 ...
bash scripts/commit.sh arm_vcpu "refactor: adapt to new axvcpu API"

# 3. 批量提交所有组件修改（在根目录执行）
cd ..
bash scripts/commit.sh axvcpu,arm_vcpu,x86_vcpu "refactor: update vCPU API and adaptations"
```

#### 3.3.5 批量代码检查

开发过程中或完成后，使用 `scripts/check.sh` 批量检查代码质量：

```bash
# 检查所有组件（ fmt、clippy、doc ）
bash scripts/check.sh all

# 检查指定组件
bash scripts/check.sh axvcpu

# 检查指定组件指定目标
bash scripts/check.sh axvcpu x86_64-unknown-none

# 在开发循环中频繁检查
# 编辑代码 → check.sh → 修复问题 → commit.sh
```

**检查项目**：
1. 代码格式（`cargo fmt`）
2. 构建检查（`cargo build`）
3. Clippy 检查（`cargo clippy`）
4. 文档构建（`cargo doc`）

**特点**：
- 一个组件失败不影响其他组件检查
- 汇总显示通过/失败列表

#### 3.3.6 批量推送组件变更

开发完成后，使用 `scripts/push.sh` 批量推送各组件的变更到远程：

```bash
# 推送单个组件
bash scripts/push.sh axvcpu

# 推送多个组件
bash scripts/push.sh arm_vcpu,axvcpu

# 推送所有有未推送提交的组件
bash scripts/push.sh all

# 推送到指定分支（默认当前分支）
bash scripts/push.sh all dev
```

**脚本功能**：
- 自动检测未推送的提交
- 自动跳过无变更的组件
- 汇总显示推送结果

#### 3.3.7 更新 submodule 引用

使用 `scripts/sync.sh` 批量同步组件到远程最新，并更新主仓库引用：

```bash
# 同步单个组件到 dev 分支最新
bash scripts/sync.sh axvcpu

# 同步多个组件
bash scripts/sync.sh arm_vcpu,axvcpu

# 同步所有组件到 dev 分支最新
bash scripts/sync.sh all

# 同步所有组件到 main 分支最新
bash scripts/sync.sh all main
```

**脚本功能**：
- 自动初始化未初始化的子模块
- 自动切换到目标分支
- 检测本地修改并提示
- 显示同步前后的 commit 变化
- 汇总显示同步结果

## 四、版本发布流程

版本发布**必须通过 axcrates 仓库统一处理**，以确保所有组件版本号保持一致。所有组件使用相同版本号，便于管理和追踪，确保组件间的依赖关系正确，并且所有组件同时发布，避免版本混乱。

| 类型 | 版本格式 | 分支 | 发布方式 | 使用场景 |
|------|----------|------|----------|----------|
| **预览版** | `0.2.0-preview.1` | dev | 直接推送 | 快速验证、内部测试 |
| **稳定版** | `0.2.0` | main | PR 合并 | 正式发布、生产使用 |

### 4.1 发布流程概述

基于 [三、开发工作流详解](#三开发工作流详解)，版本发布使用以下脚本协同完成：

```
┌─────────────────────────────────────────────────────────────────┐
│                      版本发布流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. version.sh    统一升级所有组件版本号                         │
│        │                                                        │
│        ▼                                                        │
│  2. commit.sh     提交版本变更到各组件                           │
│        │                                                        │
│        ▼                                                        │
│  3. push.sh       推送变更到远程仓库                             │
│        │                                                        │
│        ▼                                                        │
│  4. tag.sh        创建并推送版本标签                             │
│        │                                                        │
│        ▼                                                        │
│  5. publish.sh    发布到 crates.io                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 预览版发布流程

根据 [组件开发及管理规范.md](组件开发及管理规范.md) 的要求，预览版只允许在 dev 上进行发布，因此，我们需要将所有修改合并到 dev 分支后，然后按照如下流程发布预览版！

#### 4.2.1 统一升级版本号

在修改确保无误后，就可以通过 `scripts/version.sh` 统一升级组件的版本号了。

```bash
# 使用 version.sh 统一升级所有组件版本号
bash scripts/version.sh all 0.2.0-preview.1

# 验证版本号已更新
grep "^version" */Cargo.toml | head -5
```

#### 4.2.2 提交并推送版本变更

```bash
# 使用 commit.sh 批量提交所有组件的版本变更
bash scripts/commit.sh all "chore: release v0.2.0-preview.1"

# 脚本会自动：
# - 检测有版本变更的组件
# - 提交变更
# - 推送到 dev 分支
```

#### 4.2.3 创建标签

```bash
# 预览版可以选择不打标签，或打预览标签
bash scripts/tag.sh all v0.2.0-preview.1
```

#### 4.2.4 推送标签

```bash
# 预览版可以选择不打标签，或打预览标签
bash scripts/tag.sh all v0.2.0-preview.1
```

#### 4.2.5 发布到 crates.io

```bash
# 登录 crates.io
cargo login

# 使用 publish.sh 按依赖顺序发布所有组件
bash scripts/publish.sh
```

### 4.3 稳定版发布流程

根据 [组件开发及管理规范.md](组件开发及管理规范.md) 的要求，稳定版只允许在 mian（部分组件仓库使用 master 分支） 上进行发布，因此，我们需要将所有修改合并到 master 分支后，然后按照如下流程发布预览版！

#### 4.3.1 升级到稳定版版本号

在修改确保无误后，就可以通过 scripts/version.sh 统一升级组件的版本号了。

```bash
# 使用 version.sh 统一升级到稳定版版本号
bash scripts/version.sh all 0.2.0

# 验证版本号
grep "^version" */Cargo.toml | head -5
```

#### 4.3.2 提交到 main 分支

```bash
# 提交版本变更
bash scripts/commit.sh all "chore: release v0.2.0"
```

#### 4.3.3 创建 PR

在组件仓库中创建从自己使用的分支到主分支的 PR：target: main ← release-v0.2.0

#### 4.3.4 合并 PR 后打标签

等待所有 PR 合并后：

```bash
# 切换到 main 分支并更新
git checkout main
git pull --recurse-submodules

# 同步子模块到 main 最新
bash scripts/sync.sh all main

# 验证版本号统一
grep "^version = " axvcpu/Cargo.toml

# 使用 tag.sh 统一创建并推送标签
bash scripts/tag.sh all v0.2.0

# 该脚本会：
# - 检查当前分支是 main
# - 在各子模块创建标签 v0.2.0
# - 在主仓库创建标签 v0.2.0
# - 推送所有标签到远程
```

#### 4.3.5 发布到 crates.io

```bash
# 登录 crates.io
cargo login

# 使用 publish.sh 发布所有组件
bash scripts/publish.sh

# 发布顺序（自动处理）：
# Layer 0: axaddrspace, axvmconfig, axhvc, riscv-h
# Layer 1: axdevice_base, axvisor_api, axvcpu
# Layer 2: arm_vgic, x86_vlapic
# Layer 3: arm_vcpu, x86_vcpu, riscv_vcpu
# Layer 4: axdevice
# Layer 5: axvm
# Layer 6: axcrates
```

## 五、已知问题

当前基于 axcrates 统一仓库的多组件协同开发方案在多人协作场景下存在以下问题：

### 5.1 组件分支选择与合并冲突

**问题描述**：
当多个开发者同时使用 axcrates 仓库时，面临组件分支选择的困境：每个人应该选择什么分支开发？如何合并？

| 策略 | 优点 | 缺点 |
|------|------|------|
| **都用 dev 分支** | 简单统一，易于集成测试 | 同时提交时会产生 submodule 引用竞态，后推送者会覆盖先推送者的变更 |
| **每人用不同 feature 分支** | 隔离开发，不会互相干扰 | 集成困难，无法统一管理，不知道应该组合哪些分支进行测试 |

**场景1：每人在各自组件上使用不同的 feature 分支**

```bash
# Alice 开发 axvcpu 新功能
bash scripts/checkout.sh axvcpu feature/alice-api

# Bob 开发 axaddrspace 新功能  
bash scripts/checkout.sh axaddrspace feature/bob-optimize

# Carol 修复 arm_vcpu bug
bash scripts/checkout.sh arm_vcpu fix/carol-bugfix
```

**冲突情况**：

1. **axcrates 无法记录分支组合**
   - axcrates 本身只是管理仓库，不记录每个人使用了哪些分支
   - 当需要集成测试时，无法确定应该 checkout 哪些分支的组合
   - 需要人工询问每个人使用了什么分支，效率低下

2. **分支组合爆炸**
   - 5 个开发者可能使用 5 个不同的 feature 分支
   - 集成测试时需要尝试多种分支组合，难以确定正确的组合方式
   - 某个功能可能需要多个组件的分支配合，但不知道具体是哪几个

3. **集成时冲突解决复杂**
   - 当把各 feature 分支合并到 dev 时，可能产生冲突
   - 由于各组件相互依赖，一个组件的合并可能破坏其他组件的功能
   - 需要协调多个开发者同时解决冲突

**场景2：所有人都使用 dev 分支开发**

```bash
# 所有人都切换到 dev 分支
bash scripts/checkout.sh all dev

# Alice 修改 axvcpu
cd axvcpu
vim src/lib.rs
git add .
git commit -m "feat: add new API"

# Bob 修改 axaddrspace
cd ../axaddrspace
vim src/lib.rs
git add .
git commit -m "feat: optimize memory"
```

**冲突情况**：

1. **submodule 引用竞态（同一组件）**
   ```
   前提：Alice 和 Bob 都修改了 axvcpu（同一组件）
   
   时间点1: Alice 在本地提交 axvcpu (commit A)，更新 axcrates 的 submodule 引用
   时间点2: Bob 在本地提交 axvcpu (commit B)，更新 axcrates 的 submodule 引用
   时间点3: Bob 先推送 axcrates dev 分支，成功（记录 axvcpu -> commit B）
   时间点4: Alice 后推送 axcrates dev 分支，失败（需要 pull）
   时间点5: Alice pull 后 rebase，产生 submodule 引用冲突
   结果: 需要手动选择使用 commit A 还是 commit B
   
   注意：如果 Alice 和 Bob 修改的是不同组件（如 axvcpu 和 axaddrspace），
   不会互相覆盖，各自更新各自的 submodule 引用即可。
   ```

2. **频繁的强制推送**
   - 由于 submodule 引用频繁变更，每次推送都可能需要 `git pull --rebase`
   - rebase 时可能遇到 submodule 引用冲突，需要手动解决
   - 多人同时开发同一组件时，可能需要多次 rebase 才能推送成功

3. **集成测试困难**
   - 虽然都用 dev 分支，但各组件的提交时机不同
   - 当 Alice 的代码有 bug 时，会影响 Bob 基于 dev 分支的开发
   - 无法隔离测试，所有人的代码混在一起


### 5.2 Submodule 引用更新困难

**问题描述**：
axcrates 通过 `.gitmodules` 记录子模块的 commit hash。当组件不断更新时，开发人员难以确定应该使用哪个 commit hash 来更新 submodule 引用。

**核心矛盾**：
- Git Submodule 引用的是**固定的 commit hash**，而不是自动跟踪分支
- 组件仓库的 `dev` 分支在不断更新（多人提交）
- 开发人员提交组件后，需要手动更新 axcrates 中的 submodule 引用
- 但获取正确的 commit hash 非常麻烦

**场景 1：如何确定应该使用的 commit hash**

```bash
# Alice 修改了 axvcpu，在组件仓库提交并推送
cd axvcpu
git add .
git commit -m "feat: add new API"
git push origin dev
# 此时 remote dev 分支的最新 commit 是 abc1234

# Alice 回到 axcrates，需要更新 submodule 引用
cd ..
git status
# 显示：modified: axvcpu (new commits)

# 问题：Alice 应该使用哪个 commit hash？
# 选项1：cd axvcpu && git log --oneline -1 查看当前指向的 commit
# 选项2：cd axvcpu && git rev-parse HEAD 获取完整 hash
# 选项3：直接 git add axvcpu，让 git 自动记录当前的 commit
```

**问题**：
1. **选项1和2**：需要手动查看并记录 commit hash，容易出错
2. **选项3**：`git add axvcpu` 会记录当前 submodule 指向的 commit，但：
   - 如果此时其他人又提交了新的 commit，Alice 的引用又不是最新了
   - Alice 需要确保在 `git add` 之前，`axvcpu` 目录指向的是她想要的 commit

**场景 2：多人同时开发同一组件导致引用竞争**

```bash
# 初始状态：axvcpu 的 dev 分支最新 commit 是 AAAA
git submodule status axvcpu
# 输出：4a20283f8e19d03c4445346f16dc33c4f82a24b4 axvcpu (v0.2.0)

# Alice 和 Bob 各自在本地开发 axvcpu（基于同一个 AAAA）
# Alice 本地：
cd axvcpu && git commit -m "feat: alice feature"  # 本地 commit BBBB

# Bob 本地（同时，也基于 AAAA）：
cd axvcpu && git commit -m "fix: bob fix"  # 本地 commit CCCC

# Alice 先推送到 axvcpu 远程
git push origin dev  # 成功，远程 dev 最新变为 BBBB

# Bob 后推送到 axvcpu 远程
git push origin dev  # 失败！需要先 pull
git pull --rebase origin dev  # rebase 后产生新 commit CCCC'
git push origin dev  # 成功，远程 dev 最新变为 CCCC'

# 现在两人都需要更新 axcrates 的 submodule 引用
# 问题：应该指向哪个 commit？
```

**问题**：
1. **本地 commit 与远程不同步**
   - Alice 本地 axvcpu 指向 BBBB，但远程最新是 CCCC'
   - 如果 Alice 直接 `git add axvcpu`，会记录 BBBB，不是最新
   - Alice 需要先 `git pull` 更新本地，再更新 axcrates

2. **更新顺序问题**
   - 即使 Alice 更新了本地 axvcpu 到 CCCC'，在更新 axcrates 期间，可能又有新提交
   - 时间窗口问题：从 pull 到 git add 之间，远程可能有新提交

3. **submodule 引用冲突**
   - Alice 和 Bob 同时更新 axcrates 的 axvcpu 引用
   - Alice 记录的是某一时刻的 commit，Bob 记录的是另一时刻的 commit
   - 推送 axcrates 时会产生冲突，需要 rebase 并重新确定引用

**场景 3：使用 axcrates 中的 hash 还是组件仓库的 hash**

```bash
# Alice 想确认 axvcpu 在 axcrates 中应该指向哪个 commit

# 方法1：查看 axcrates 中当前记录的 hash
git ls-tree HEAD axvcpu
# 输出：160000 commit 4a20283f8e19d03c4445346f16dc33c4f82a24b4	axvcpu

# 方法2：进入组件仓库查看
cd axvcpu
git log --oneline -5
# 输出：
# a1b2c3d feat: latest feature
# b2c3d4e fix: previous fix
# c3d4e5f feat: another feature
# ...

# 方法3：查看远程最新
git log origin/dev --oneline -1
# 输出：a1b2c3d feat: latest feature
```

**矛盾**：
- axcrates 中记录的 hash（方法1）可能是旧的
- 组件仓库的日志（方法2）显示本地可能有多个 commit
- 远程最新（方法3）才是真正的最新
- 开发人员不知道应该以哪个为准

### 5.3 版本号管理冲突

**问题描述**：
使用 `scripts/version.sh` 统一升级版本号时：

**场景 1：并发版本升级**
- Alice 升级版本到 `0.3.0-preview.1` 并提交
- Bob 同时升级版本到 `0.3.0-preview.1` 并提交
- 后推送者需要解决版本号冲突

**场景 2：预览版版本混乱**
- dev 分支版本：`0.3.0-preview.1`
- Alice 需要发布测试版本，升级到 `0.3.0-preview.2`
- Bob 也需要发布测试版本，升级到 `0.3.0-preview.3`
- 频繁的预览版升级导致版本号快速增长

**建议解决方案**：
- 指定专人（Release Manager）负责版本发布
- 使用锁定机制（如 GitHub Branch Protection）防止多人同时修改版本号
- 建立版本升级的预约/协调机制

### 5.4 稳定版发布工作量巨大

**问题描述**：
当前架构下，发布稳定版本需要为每个组件仓库单独创建 PR 并等待合并。当组件数量较多时，工作量呈线性增长。

**场景：发布包含 14 个组件的稳定版本**

```
步骤 1：升级版本号
  bash scripts/version.sh all 0.2.0
  → 修改 14 个组件的 Cargo.toml

步骤 2：为每个组件创建 PR（需要在 GitHub 上操作 14 次）
  组件仓库 PR 列表：
  - axvcpu:      feature/release-v0.2.0 → main
  - axaddrspace: feature/release-v0.2.0 → main
  - arm_vcpu:    feature/release-v0.2.0 → main
  - arm_vgic:    feature/release-v0.2.0 → main
  - axdevice:    feature/release-v0.2.0 → main
  - ... (共 14 个 PR)

步骤 3：等待所有 PR 合并
  - 每个 PR 需要 review
  - 可能需要修复 CI 问题
  - 部分组件可能被阻塞

步骤 4：更新 axcrates 的 submodule 引用
  git submodule update --remote
  git add .
  git commit -m "chore: update submodules to v0.2.0"
  → 又需要创建 PR

步骤 5：打标签并发布
  bash scripts/tag.sh all v0.2.0
  bash scripts/publish.sh
```

**工作量估算**：

| 组件数量 | 需创建的 PR | 需等待合并 | 手动操作次数 |
|---------|------------|-----------|-------------|
| 14 个（当前） | 15 个（14 组件 + 1 主仓库） | 15 次 | ~30 次 |
| 50 个 | 51 个 | 51 次 | ~100 次 |
| 100 个 | 101 个 | 101 次 | ~200 次 |

**问题**：
1. **手动操作繁琐**：需要在 GitHub 上为每个组件手动创建 PR
2. **等待时间长**：需要等待所有 PR 逐一 review 和合并
3. **串行依赖**：axcrates 的 PR 必须等所有组件 PR 合并后才能创建
4. **容错性差**：任何一个组件 PR 被阻塞，整个发布流程停滞
5. **难以自动化**：由于分支保护，无法通过脚本自动合并
