## 一、方案概述

### 1.1 背景与目的

ArceOS Hypervisor 采用**组件化架构设计**，组件是功能相对独立且可独立测试的实体。为了规范组件的开发、管理和发布流程，特制定本协同开发方案。

### 1.2 仓库架构

我们的组件**实际分布在各个独立的子仓库中**，每个组件都是一个独立的 Git 仓库。同时，我们提供了 **axcrates** 仓库作为统一的管理入口，通过 Git Submodule 关联所有组件。**因此，有两种开发方式可选，来统一开发组件：**

| 方式 | 适用场景 | 特点 |
|------|----------|------|
| **独立子仓库开发** | 单组件修改 | 直接克隆子仓库，轻量快捷 |
| **axcrates 统一仓库开发** | 跨组件协作 | 通过主仓库统一管理所有组件 |

### 1.3 架构关系图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         组件分布与协作架构                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   独立子仓库（14个组件）                    axcrates（统一管理仓库）     │

│   │ arm_vcpu     │◄───────────────────────►│  .gitmodules         │    │
│   │ arm_vgic     │◄───────────────────────►│  scripts/            │    │
│   │ axaddrspace  │◄───────────────────────►│  ├── check.sh        │    │
│   │ axdevice     │◄───────────────────────►│  ├── version.sh      │    │
│   │ axdevice_base│◄───────────────────────►│  └── publish.sh      │    │
│   │ axhvc        │◄───────────────────────►│                      │    │

│   │ axvmconfig   │◄───────────────────────►│  axaddrspace/        │    │

│   • 方式一：直接克隆子仓库（如 arm_vcpu）独立开发                        │

│                                                                         │

```

**架构说明：**
- **独立子仓库**：每个组件（如 arm_vcpu、axvcpu）都有自己的独立 Git 仓库，可以独立开发、测试、发布
- **axcrates 主仓库**：通过 Git Submodule 引用所有组件，提供统一的版本管理和协同脚本
- **.gitmodules**：记录每个子模块的具体 commit，确保版本可追溯和可复现
- **scripts/**：提供批量操作脚本，如批量升级版本号、批量发布、同步子模块等

### 1.4 核心开发原则

基于 [组件开发及管理规范.md](组件开发及管理规范.md)，我们遵循以下核心原则：

| 原则 | 说明 | 实践方式 |
|------|------|----------|
| **版本统一** | 所有组件使用相同版本号，便于管理和追踪 | `version.sh` 批量管理所有组件版本号 |
| **分支隔离** | main 分支保护（需 PR），dev 分支开放（可直接推送） | PR 流程 vs 直接推送 |
| **依赖锁定** | 主仓库记录子模块具体 commit，确保可复现构建 | `.gitmodules` + submodule |
| **独立演进** | 子仓库可独立开发测试，不受主仓库影响 | 直接克隆子仓库开发 |
| **代码质量** | 所有组件必须通过 CI 检查（fmt、clippy、doc、test） | `scripts/check.sh` 本地检查 |
| **文档完整** | 公开发布的组件必须具备完整的 rustdoc 文档 | `cargo doc` 生成文档 |

### 1.5 分支与版本对应关系

在 [组件开发及管理规范.md](组件开发及管理规范.md) 中我们规定了，组件仓库分支与版本必须强制对应，以便统一管理！

## 二、开发环境准备

### 2.1 配置 Git 环境

```bash
# 配置全局用户名和邮箱（如未配置）
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# 配置 SSH 密钥（用于访问 GitHub）
# 如已有密钥可跳过
ssh-keygen -t ed25519 -C "your.email@example.com"
cat ~/.ssh/id_ed25519.pub
# 将公钥添加到 GitHub: Settings -> SSH and GPG keys -> New SSH key
```

### 2.2 安装 Rust 工具链

```bash
# 安装 Rust（如已安装可跳过）
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source $HOME/.cargo/env

# 安装所需目标架构
rustup target add aarch64-unknown-none-softfloat
rustup target add x86_64-unknown-none
rustup target add riscv64gc-unknown-none-elf

# 安装必要组件
rustup component add rust-src llvm-tools rustfmt clippy
```

## 三、开发工作流详解

### 3.1 工作流选择决策树

```
开始开发
    │
    ├──► 只修改单个组件？
    │       │
    │       ├──► 是 ──► 选择模式 A：独立子仓库开发
    │       │
    │       └──► 否 ──► 需要修改多个组件？
    │               │
    │               ├──► 是 ──► 选择模式 B：主仓库协同开发
    │               │
    │               └──► 否 ──► 只需要测试？
    │                       │
    │                       ├──► 是 ──► 选择模式 B：主仓库协同开发
    │                       │
    │                       └──► 否 ──► 咨询维护者
    │
    └──► 发布版本？
            │
            ├──► 预览版 ──► 使用 version.sh + commit.sh + tag.sh
            │
            └──► 稳定版 ──► 使用 version.sh + commit.sh + tag.sh + PR
```

### 3.2 独立子组件开发

**适用场景**：
- 只修改单个组件
- 不涉及其他组件的 API 变更

#### 3.2.1 克隆子仓库

```bash
# 以 axvcpu 为例
git clone git@github.com:arceos-hypervisor/axvcpu.git
cd axvcpu

# 查看当前分支
git branch -a
```

#### 3.2.2 选择开发分支

```bash
# 情况 1：新功能开发（推送到 dev 分支）
git checkout dev
git pull origin dev

# 情况 2：补丁修复（推送到 main 分支，需 PR）
git checkout main
git pull origin main
```

#### 3.2.3 创建功能分支

```bash
# 创建功能分支（即使 dev 分支可直接推送，也建议使用功能分支）
git checkout -b feature/add-new-api

# 分支命名规范：
# - feature/xxx  - 新功能
# - fix/xxx      - Bug 修复
# - refactor/xxx - 代码重构
# - docs/xxx     - 文档更新
```

#### 3.2.4 开发与本地测试

```bash
# 编辑代码...

# 格式化代码
cargo fmt

# 静态检查
cargo clippy --all-targets --all-features -- -D warnings

# 运行测试
cargo test --all-features

# 构建文档
cargo doc --no-deps
```

#### 3.2.5 提交变更

```bash
# 查看变更
git status
git diff

# 添加变更
git add .

# 提交（使用约定式提交格式）
git commit -m "feat: add new virtualization API

This commit adds support for new vCPU features:
- Add vCPU state save/restore API
- Support for new interrupt handling

Closes #123"

# 推送
git push origin feature/add-new-api
```

#### 3.2.6 创建 Pull Request

**情况 1：推送到 dev 分支（预览版迭代）**
```
GitHub 操作：
1. 打开 https://github.com/arceos-hypervisor/axvcpu
2. 点击 "Compare & pull request"
3. base: dev ← compare: feature/add-new-api
4. 填写 PR 标题和描述
5. 请求 review（可选，dev 分支可快速合并）
6. 合并 PR
```

**情况 2：推送到 main 分支（稳定版）**
```
GitHub 操作：
1. 打开 https://github.com/arceos-hypervisor/axvcpu
2. 点击 "Compare & pull request"
3. base: main ← compare: feature/add-new-api
4. 填写详细的 PR 描述
5. 确保 CI 检查通过
6. 请求至少 1 人 review
7. 合并 PR（需 maintainer 权限）
```

#### 3.2.7 通知主仓库更新

```bash
# PR 合并后，通知主仓库维护者更新 submodule
# 提供以下信息：
# - 组件名称：axvcpu
# - 更新的分支：dev 或 main
# - 变更简述：添加了新的 vCPU API
```

### 3.3 多组件协同开发

多组件协同开发适用于需要同时修改多个组件，或者跨组件 API 变更等情况。为此，我们创建了 https://github.com/arceos-hypervisor/axcrates 统一仓库，通过 SUBMODULE 来管理所有组件。

与独立开发的最大区别，axcrates 提供了一系列**协同开发脚本**，可以**统一管理所有组件**，无需进入每个组件目录执行 git 命令。
| 脚本 | 功能 | 示例 |
|------|------|------|
| `checkout.sh` | 批量切换分支 | `bash scripts/checkout.sh all dev` |
| `check.sh` | 批量代码检查 | `bash scripts/check.sh all` |
| `commit.sh` | 批量提交组件 | `bash scripts/commit.sh axvcpu "feat: xxx"` |
| `push.sh` | 批量推送组件 | `bash scripts/push.sh all` |
| `sync.sh` | 批量同步子模块 | `bash scripts/sync.sh all dev` |
| `version.sh` | 批量更新版本 | `bash scripts/version.sh all 0.2.0` |
| `tag.sh` | 批量创建标签 | `bash scripts/tag.sh all v0.2.0` |

**使用建议**：
- **日常开发**：使用 `checkout.sh` 统一管理分支，`check.sh` 批量检查代码
- **提交推送**：使用 `commit.sh` 批量提交，`push.sh` 批量推送
- **版本发布**：使用 `version.sh` 统一升级版本，`tag.sh` 批量创建标签
- **保持同步**：定期使用 `sync.sh` 同步所有组件到最新

#### 3.3.1 获取 axcrates 仓库

```bash
# 克隆主仓库（包含所有子模块）
git clone --recurse-submodules git@github.com:arceos-hypervisor/axcrates.git
cd axcrates

# 切换到 dev 分支
git checkout dev
```
#### 3.3.2 批量切换组件分支

使用 `scripts/checkout.sh` 批量管理组件分支：

```bash
# 切换所有组件到 dev 分支
bash scripts/checkout.sh all dev

# 切换所有组件到 main 分支
bash scripts/checkout.sh all main

# 切换单个组件到 feature 分支
bash scripts/checkout.sh axvcpu feature/new-api
```

**脚本功能**：
- 自动检测当前分支，如在目标分支则跳过
- 自动从远程获取并创建跟踪分支
- 检测未提交变更并提示
- 汇总显示切换结果

#### 3.3.3 确定修改范围

```bash
# 分析依赖关系，确定需要修改的组件
# 示例：修改 axvcpu API，影响 arm_vcpu 和 x86_vcpu

# 查看组件依赖树
cd axvcpu && cargo tree
cd ../arm_vcpu && cargo tree | grep axvcpu
cd ../x86_vcpu && cargo tree | grep axvcpu
```

#### 3.3.4 修改组件代码

开发阶段进入各组件目录修改代码。此时所有组件已在 dev 分支（由 3.3.2 设置），开发流程与 3.2 独立子组件开发类似，但是可使不用在组件中直接提交代码，而是在编辑多个组件之后，统一使用 `check.sh`、`commit.sh` 批量处理多个组件。

**单组件处理**：

```bash
# 进入组件目录
cd axvcpu

# ... 编辑代码 ...
# ... 测试 ...
# ... CHECK ...

# 使用 commit.sh 提交并推送（推荐）
bash scripts/commit.sh axvcpu "refactor: update vCPU API for better abstraction"

# 或者手动提交
git add .
git commit -m "refactor: update vCPU API for better abstraction"
git push origin feature/refactor-api
```

**多组件批量处理**：

```bash
# 场景：修改了 axvcpu API，需要适配 arm_vcpu 和 x86_vcpu

# 1. 在 axvcpu 开发并提交
cd axvcpu
# ... 编辑代码 ...
# ... 测试 ...
# ... CHECK ...
# ... 提交 ...
bash scripts/commit.sh axvcpu "refactor: update vCPU API"

# 2. 在 arm_vcpu 适配并提交
cd ../arm_vcpu
# ... 编辑代码 ...
# ... 测试 ...
# ... CHECK ...
# ... 提交 ...
bash scripts/commit.sh arm_vcpu "refactor: adapt to new axvcpu API"

# 3. 批量提交所有组件修改（在根目录执行）
cd ..
bash scripts/commit.sh axvcpu,arm_vcpu,x86_vcpu "refactor: update vCPU API and adaptations"
```

#### 3.3.5 批量代码检查

开发过程中或完成后，使用 `scripts/check.sh` 批量检查代码质量：

```bash
# 检查所有组件（ fmt、clippy、doc ）
bash scripts/check.sh all

# 检查指定组件
bash scripts/check.sh axvcpu

# 检查指定组件指定目标
bash scripts/check.sh axvcpu x86_64-unknown-none

# 在开发循环中频繁检查
# 编辑代码 → check.sh → 修复问题 → commit.sh
```

**检查项目**：
1. 代码格式（`cargo fmt`）
2. 构建检查（`cargo build`）
3. Clippy 检查（`cargo clippy`）
4. 文档构建（`cargo doc`）

**特点**：
- 一个组件失败不影响其他组件检查
- 汇总显示通过/失败列表

#### 3.3.6 批量推送组件变更

开发完成后，使用 `scripts/push.sh` 批量推送各组件的变更到远程：

```bash
# 推送单个组件
bash scripts/push.sh axvcpu

# 推送多个组件
bash scripts/push.sh arm_vcpu,axvcpu

# 推送所有有未推送提交的组件
bash scripts/push.sh all

# 推送到指定分支（默认当前分支）
bash scripts/push.sh all dev
```

**脚本功能**：
- 自动检测未推送的提交
- 自动跳过无变更的组件
- 汇总显示推送结果

#### 3.3.7 更新 submodule 引用

使用 `scripts/sync.sh` 批量同步组件到远程最新，并更新主仓库引用：

```bash
# 同步单个组件到 dev 分支最新
bash scripts/sync.sh axvcpu

# 同步多个组件
bash scripts/sync.sh arm_vcpu,axvcpu

# 同步所有组件到 dev 分支最新
bash scripts/sync.sh all

# 同步所有组件到 main 分支最新
bash scripts/sync.sh all main
```

**脚本功能**：
- 自动初始化未初始化的子模块
- 自动切换到目标分支
- 检测本地修改并提示
- 显示同步前后的 commit 变化
- 汇总显示同步结果

## 四、版本发布流程

版本发布**必须通过 axcrates 仓库统一处理**，以确保所有组件版本号保持一致。所有组件使用相同版本号，便于管理和追踪，确保组件间的依赖关系正确，并且所有组件同时发布，避免版本混乱。

| 类型 | 版本格式 | 分支 | 发布方式 | 使用场景 |
|------|----------|------|----------|----------|
| **预览版** | `0.2.0-preview.1` | dev | 直接推送 | 快速验证、内部测试 |
| **稳定版** | `0.2.0` | main | PR 合并 | 正式发布、生产使用 |

### 4.1 发布流程概述

基于 [三、开发工作流详解](#三开发工作流详解)，版本发布使用以下脚本协同完成：

```
┌─────────────────────────────────────────────────────────────────┐
│                      版本发布流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. version.sh    统一升级所有组件版本号                         │
│        │                                                        │
│        ▼                                                        │
│  2. commit.sh     提交版本变更到各组件                           │
│        │                                                        │
│        ▼                                                        │
│  3. push.sh       推送变更到远程仓库                             │
│        │                                                        │
│        ▼                                                        │
│  4. tag.sh        创建并推送版本标签                             │
│        │                                                        │
│        ▼                                                        │
│  5. publish.sh    发布到 crates.io                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 预览版发布流程

**目标**：发布 `0.2.0-preview.1` 供团队内部测试，注意预览版只能在 dev 分支上发布！

#### 4.2.1 统一升级版本号

在修改确保无误后，就可以通过 scripts/version.sh 统一升级组件的版本号了。

```bash
# 使用 version.sh 统一升级所有组件版本号
bash scripts/version.sh all 0.2.0-preview.1

# 验证版本号已更新
grep "^version" */Cargo.toml | head -5
```

#### 4.2.2 提交并推送版本变更

```bash
# 使用 commit.sh 批量提交所有组件的版本变更
bash scripts/commit.sh all "chore: release v0.2.0-preview.1"

# 脚本会自动：
# - 检测有版本变更的组件
# - 提交变更
# - 推送到 dev 分支
```

#### 4.2.3 创建标签

```bash
# 预览版可以选择不打标签，或打预览标签
bash scripts/tag.sh all v0.2.0-preview.1
```

#### 4.2.4 推送

```bash
# 预览版可以选择不打标签，或打预览标签
bash scripts/tag.sh all v0.2.0-preview.1
```

#### 4.2.5 发布到 crates.io

### 4.3 稳定版发布流程

**目标**：发布 `0.2.0` 正式版本

#### 4.3.1 升级到稳定版版本号

在修改确保无误后，就可以通过 scripts/version.sh 统一升级组件的版本号了。

```bash
# 使用 version.sh 统一升级到稳定版版本号
bash scripts/version.sh all 0.2.0

# 验证版本号
grep "^version" */Cargo.toml | head -5
```

#### 4.3.2 提交到 dev 分支

```bash
# 提交版本变更
bash scripts/commit.sh all "chore: release v0.2.0"
```

#### 4.3.3 创建 Release 分支并提 PR

```bash
# 在各组件创建 release 分支
cd axvcpu && git checkout -b release-v0.2.0 && git push origin release-v0.2.0
cd ../axaddrspace && git checkout -b release-v0.2.0 && git push origin release-v0.2.0
# ... 其他组件

# 在主仓库创建 release 分支
cd ..
git checkout -b release-v0.2.0
git add .
git commit -m "chore: release v0.2.0"
git push origin release-v0.2.0
```

然后在 GitHub 上创建 PR（target: main ← release-v0.2.0）。

#### 4.3.4 合并 PR 后打标签

等待所有 PR 合并后：

```bash
# 切换到 main 分支并更新
git checkout main
git pull --recurse-submodules

# 同步子模块到 main 最新
bash scripts/sync.sh all main

# 验证版本号统一
grep "^version = " axvcpu/Cargo.toml

# 使用 tag.sh 统一创建并推送标签
bash scripts/tag.sh all v0.2.0

# 该脚本会：
# - 检查当前分支是 main
# - 在各子模块创建标签 v0.2.0
# - 在主仓库创建标签 v0.2.0
# - 推送所有标签到远程
```

#### 4.3.5 发布到 crates.io

```bash
# 登录 crates.io
cargo login

# 使用 publish.sh 发布所有组件
bash scripts/publish.sh

# 发布顺序（自动处理）：
# Layer 0: axaddrspace, axvmconfig, axhvc, riscv-h
# Layer 1: axdevice_base, axvisor_api, axvcpu
# Layer 2: arm_vgic, x86_vlapic
# Layer 3: arm_vcpu, x86_vcpu, riscv_vcpu
# Layer 4: axdevice
# Layer 5: axvm
# Layer 6: axcrates
```
